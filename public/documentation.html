Présentation du projet

Mon Site Cosmétique est une plateforme e-commerce complète permettant la gestion de produits, commandes, utilisateurs et livraisons, avec des dashboards dédiés pour chaque rôle (admin, vendeur, livreur, client).

  • Gestion multi-rôles (admin, vendeur, livreur, client)
  • Statistiques avancées, reporting, notifications
  • Interface responsive et moderne

Pourquoi ce projet est-il pertinent ?

  • Pertinence technique : Utilisation de technologies modernes (Laravel, React, Spatie, Tailwind), gestion avancée des rôles, sécurité, tests, notifications, etc.
  • Pertinence professionnelle : Reflète les besoins réels d'une plateforme e-commerce multi-profils, idéal pour le portfolio ou la préparation au monde du travail.
  • Pertinence pédagogique : Permet d'apprendre le cycle complet d'un projet web (backend, frontend, base de données, sécurité, UX/UI, débogage, migration de données, etc.).
  • Pertinence métier : Le secteur de la cosmétique et de l'e-commerce est en pleine croissance, avec des besoins concrets en gestion de rôles, reporting, alertes, etc.
  • Pertinence pour la veille et l'innovation : Découverte de packages avancés, bonnes pratiques, ouverture à l'intégration de services externes, évolutivité du projet.

Ce projet constitue une base solide pour un mémoire, un portfolio ou une évolution vers des fonctionnalités encore plus avancées (API, mobile, CI/CD, etc.).

🔒 Sécurité et Isolation des Données

Middleware de Sécurité

  • IsVendeur - Vérification du rôle vendeur
  • IsAdmin - Vérification du rôle admin
  • Filtrage automatique par seller_id
  • Protection contre l'accès croisé

Isolation des Données

  • Chaque vendeur ne voit que ses produits
  • Chaque vendeur ne voit que ses commandes
  • Chaque vendeur ne voit que ses catégories
  • Impossible d'accéder aux données d'autres vendeurs

🛡️ Exemple de Code Sécurisé

// Dans VendorProductController
public function index()
{
    $user = auth()->user();
    
    if ($user->hasRole('vendeur')) {
        // Les vendeurs ne voient que leurs produits
        $products = Product::where('seller_id', $user->id)->paginate(10);
    } elseif ($user->hasRole('admin')) {
        // Les admins voient tous les produits
        $products = Product::with('seller')->paginate(10);
    } else {
        abort(403, 'Accès non autorisé.');
    }
    
    return view('vendor.products.index', compact('products'));
}

Notifications & logique

  • Mail : confirmation commande, bienvenue vendeur, blocage/déblocage, alerte stock bas envoyée à l'admin ET au vendeur concerné, notification nouvelle commande à chaque vendeur impliqué.
  • In-app : nouvelle commande, changement de statut, stock bas, notification admin, notifications contextuelles pour chaque vendeur.
  • Push (optionnel) : à activer si besoin

Chaque notification est envoyée selon le rôle, l’action et le contexte métier. Les vendeurs reçoivent toutes les notifications opérationnelles liées à leurs produits et commandes, mais jamais les notifications sensibles réservées à l'admin.

    Exemple (User.php) : 
    public $name; // +name : string
    Le nom d’un utilisateur est accessible dans toute l’application (affichage, notifications, etc.).
  • - (private) : accessible uniquement dans la classe.
    Exemple (User.php) : 
    private $password; // -password : string
    Le mot de passe ne doit être accessible que dans la classe User, jamais à l’extérieur (sécurité).
  • # (protected) : accessible dans la classe et ses sous-classes.
    Exemple (User.php ou classe héritée) : 
    protected function hashPassword($password) { ... } // #hashPassword(string): string
    Cette méthode peut être utilisée dans User et dans ses sous-classes (ex : AdminUser), mais pas ailleurs.
  • ~ (package) : accessible dans le même package/module (rare en PHP, courant en Java).
    Exemple (Java) : 
    String sessionToken; // ~sessionToken : String
    Attribut accessible uniquement dans le même package/module.

Cardinalités (multiplicités)

  • 1 : exactement un.
    Exemple : 
    User "1" --o "many" Order : passe
    Un utilisateur passe plusieurs commandes, mais chaque commande appartient à un seul utilisateur.
  • 0..1 : zéro ou un.
    Exemple : 
    Product {
  +int? seller_id // 0 ou 1 vendeur associé
}
    Un produit peut être associé à un vendeur (marketplace) ou non (produit admin).
  • * : plusieurs.
    Exemple : 
    Vendeur "1" --o "*" Product : possède
    Un vendeur possède plusieurs produits.
  • 1..* : au moins un.
    Exemple : 
    Order "1" --* "1..*" Product : contient
    Une commande contient au moins un produit.

Relations entre classes

  • -- : association simple.
    Exemple : 
    User -- Order
    Un utilisateur est lié à ses commandes.
  • --|> : héritage/généralisation.
    Exemple : 
    User --|> Admin
    Admin hérite de User (héritage de rôle ou de méthodes communes).
  • --o : agrégation.
    Exemple : 
    Vendeur --o Product
    Un vendeur agrège des produits, mais un produit peut exister sans vendeur (produit admin).
  • --* : composition.
    Exemple : 
    Order --* OrderLine
    Une ligne de commande n’existe que dans une commande (composition forte).

Stéréotypes UML

  • <<interface>> : interface (contrat, pas d’implémentation).
    Exemple : 
    <<interface>> NotificationChannel
    Interface pour les canaux de notification (mail, in-app, SMS, etc.).
  • <<abstract>> : classe abstraite (ne peut pas être instanciée).
    Exemple : 
    <<abstract>> BaseController
    Classe de base pour tous les contrôleurs Laravel, non instanciable directement.

Types d'attributs

  • int : entier.
    Exemple : 
    Product {
  +int quantity
}
    La quantité d’un produit en stock.
  • string : texte.
    Exemple : 
    User {
  +string email
}
    L’email d’un utilisateur.
  • float : nombre décimal.
    Exemple : 
    Product {
  +float price
}
    Le prix d’un produit.
  • bool : booléen (vrai/faux).
    Exemple : 
    Product {
  +bool is_active
}
    Indique si le produit est visible à la vente.
  • timestamp : date/heure.
    Exemple : 
    Order {
  +timestamp created_at
}
    Date de création de la commande.

Valeurs typiques d'attributs (exemples projet)

  • status (Order) : en attente, payé, expédié, livré, annulé…
    Décrit l’état d’une commande à chaque étape.
    Exemple d’utilisation dans Order.php :     
    $order->status = 'expédié';
  • role (User) : admin, vendeur, livreur, client…
    Définit les droits et accès de l’utilisateur.
    Exemple d’utilisation dans User.php :     
    if ($user->role === 'vendeur') { ... }
  • is_active (Product) : true ou false
    Indique si le produit est visible à la vente.
    Exemple d’utilisation dans Product.php :     
    if ($product->is_active) { ... }

Quand utiliser chaque sigle ?

  • Visibilité : toujours préciser pour chaque attribut/méthode dans un diagramme de classe.
  • Cardinalité : à chaque relation entre classes (ex : 1 User --* Order)
  • Relations : pour montrer la nature du lien (héritage, composition, etc.)
  • Stéréotype : si la classe est abstraite ou une interface
  • Type : pour chaque attribut, pour la clarté
  • Valeurs : utile dans la doc pour expliquer le métier, pas forcément dans le diagramme

Pour aller plus loin, voir la documentation UML officielle.
